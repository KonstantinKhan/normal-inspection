# Рефакторинг файлового аплоада — саммари

## Блок 1. От пустого try-catch к осмысленной обработке ошибок

**Стартовая точка:** `FileStorage.store` содержал try-catch, который только перебрасывал исключение (`catch (e) { throw e }`) — нулевая ценность, вводит в заблуждение.

**Принцип:** try-catch оправдан, когда внутри есть **действие** — логирование, оборачивание в доменный тип, cleanup. Пустой rethrow = удалить.

**Ключевой инсайт:** наличие sealed `FileResult` говорило, что автор **хотел** кодировать исход в типе, но не довёл. Правильный финиш — ловить ошибку и возвращать доменный `Failure`, а не пробрасывать.

---

## Блок 2. Корутины и `CancellationException`

**Главное правило:** `CancellationException` — **нормальный control flow** structured concurrency, не ошибка. Возникает штатно (timeout, отмена Job, пользователь ушёл). Никогда не глотать, никогда не логировать на ERROR/WARN.

**Антипаттерны:**
- `catch (Exception)` / `catch (Throwable)` в `suspend` → глотает `CancellationException` → корутина «переживает» свою отмену.
- `runCatching` ловит `Throwable` → тот же эффект + ловит `Error` (OOM).
- Логировать cancellation → alert fatigue, шум топит реальные ошибки.

**Правильный паттерн:**
```kotlin
} catch (e: CancellationException) {
    throw e                          // проброс, без лога
} catch (e: IOException) {           // узкий тип
    logger.error(...)
    ...
}
```

**Принцип выбора catch:** лови **узкий** тип (`IOException`), а не `Exception`. Узкий catch физически не поймает `CancellationException`/`Error` — ошибку сделать невозможно.

## Блок 3. `try-catch` vs `runCatching` — когда что

**Принципиальная разница:**
- `try-catch` — **управляющая конструкция**, ветвление в точке сбоя.
- `runCatching` — **превращает исключение в данные** (`kotlin.Result<T>`), функциональная композиция.

**Дерево решений:**
- Код в корутине / нужен `CancellationException`-проброс → `try-catch`.
- Доменный sealed-result уже есть → `try-catch`, маппь в домен.
- Разные исключения → разная обработка → `try-catch` (несколько catch-веток).
- Нужен `finally` для cleanup → `try-catch`.
- Синхронный код, чистые вычисления, return-тип `kotlin.Result` → `runCatching`.

**Для нашего случая (suspend + доменный result) — однозначно `try-catch`.** Можно «починить» runCatching перевыбросом в `onFailure`, но это борьба с инструментом: широкий `Throwable` + ручной двойной фильтр + лишняя обёртка.

## Блок 4. Логирование

**Что логировать:** `IOException` (реальный сбой) на ERROR с контекстом + cause. `CancellationException` — **никогда** (штатный flow).

**Как подключить:** SLF4J через `LoggerFactory.getLogger(...)` — `logback-classic` уже в deps, новая зависимость не нужна. Параметризованный лог: `logger.error("msg: {}", arg, exception)` (SLF4J трактует последний Throwable как cause).

**Оговорка:** SLF4J не принимает lambda (`logger.debug { ... }` — это kotlin-logging). Поймал реальную ошибку компиляции при gate-сборке → перевёл на параметризованный SLF4J.

**Уровни:** ERROR = «дежурный встаёт», WARN = «глянуть в рабочий день», DEBUG/TRACE = диагностика. Cancellation = ничего.

## Блок 5. От «чистого кода» к «продакшн-стораджу»

**Чистый код ≠ прод-код.** Обсуждённая обработка ошибок была сеньорской, но как **боевой сторадж** имели дыры:

| Риск | Решение |
|---|---|
| Неатомарная запись (частичный файл при сбое) | temp-файл + `Files.move(ATOMIC_MOVE)` |
| Path injection (`../`-traversal) | санитизация имени / base-dir + normalize |
| `InputStream` не закрывается | `.use {}` у владельца (где поток рождён) |
| Нет лимита размера → DoS | `LimitedInputStream` |
| `println` в прод-коде | logger |
| Дублирующий `withContext(Dispatchers.IO)` | убрать (у `store` уже есть) |
| Public mutable `val storage` | `private` + DI через конструктор |

**Принцип:** прод-код отличается тем, что закрывает **невидимые** риски с данными (целостность, race, обход директорий, переполнение), а не только красиво ловит исключения.

## Блок 6. Владение ресурсами — кто открыл, тот закрыл

**Трассировка потока:** рождается в `FileRouter` (`part.provider().toInputStream()`) → пробрасывается вниз через `FileReceiver` → потребляется в `FileStorage.Files.copy`. Никто не закрывал → утечка.

**Принцип RAII:** закрывает **владелец** — там, где ресурс создан. Не потребитель. Потребитель, закрывающий чужой ресурс → скрытый контракт, риск double-close/use-after-close.

**Фикс:** `.use { }` в `FileRouter` вокруг создания потока. `use` закрывает при любом исходе (успех/исключение/отмена).

**Пойманный по ходу баг в плане:** изначально собирал `PartData.FileItem` в список и звал `provider()` **после** `part.release()` — это убило бы чтение. Исправил: обработка **внутри** `forEachPart`, до `release`.

## Блок 7. Слои и доменные типы — где живёт `FileResult`

**Запах:** один `FileResult` обслуживала **три слоя** одновременно:
- `FileStorage` (инфра) создавал `Success`/`Failure`
- `FileRouter` (транспорт) создавал `FileNotProvided`
- `FileReceiver` (домен, в плане) создавал бы `Rejected`

Три создателя на трёх уровнях для одного типа = смешение ответственностей. Не «3 модуля используют тип» (это норм — для этого `common` и есть), а «3 слоя с разной семантикой мутируют один тип».

**Лекарство — разделение по слоям, не по модулям:**

| Тип | Слой | Что кодирует |
|---|---|---|
| `StorageOutcome` (`Written`/`Failed`) | Инфра | «записано / диск упал» — не знает бизнес-причин |
| `ReceptionResult` (`Accepted`/`Rejected`) | Домен | бизнес-исход приёма файла |
| HTTP-mapping | Транспорт | маппит `ReceptionResult` → статус-коды |

**Перевод на границе:** `FileReceiver` переводит `StorageOutcome → ReceptionResult`. Инфра не знает доменных понятий.

**«Не передан» ушёл из домена:** `NotProvided` — **транспортный факт** (нет file-part в multipart). Роутер коротит `400` сам, **до** вызова ресивера. Домен получает только реальные файлы. Контракт ресивера: «вот файл, прими».

**Принцип:** домен не должен знать слово «отсутствие». Транспортные предусловия enforce'ятся на транспортном слое.

## Блок 8. Безопасность — `sanitizeName`

**Угроза:** `originalFileName` из multipart **полностью контролируется клиентом** (`Content-Disposition`). Без санитизации → **arbitrary file write** (один из опаснейших классов → часто RCE).

**Кейсы атаки:** `../../etc/cron.d/pwn` (RCE через cron), `/var/www/html/shell.jsp` (webshell), `..\..\Windows\evil.dll`, перезапись конфигов/`.bashrc`/`authorized_keys`.

**Механизм:** `substringAfterLast('/').substringAfterLast('\\').trim()` — оставляет только basename. Любой traversal коллапсирует в один файл в CWD, эскейп невозможен.

**Предел:** фиксит traversal, но не запись в CWD и не коллизии. Усиление: `baseDir` + `Path(base).resolve(name).normalize()` + `startsWith(base)` (defense in depth) + стратегия уникальных имён.

## Блок 9. Безопасность — `LimitedInputStream` (DoS-гард)

**Угроза:** unbounded upload → переполнение диска/памяти → DoS. `Content-Length` ненадёжен (может отсутствовать, клиент врёт).

**Механизм:** обёртка над `InputStream`, считает реально прочитанные байты в кумулятивный счётчик `read: Long`, бросает `SizeLimitExceededException` при `read > maxBytes`. `Files.copy` блочно читает (8192) → ~12 800 чанков на 100 MiB → на 12 801-м сумма переваливает → аборт.

**Ключевое решение — `RuntimeException`, не `IOException`:** чтобы сигнал `TOO_LARGE` **пролетел мимо** узкого `IOException`-catch в `store` и добрался до ресивера, где классифицируется как доменный отказ. Если бы был `IOException` → store обернул бы в `StorageOutcome.Failed` → слилось бы с «диск упал» → потеря семантики.

**Оверхед:** ~2-4 ns на чанк vs микросекунды I/O → <0.1%, JIT инлайнит. Памяти не жрёт (pass-through). Стоит того.

**Нюансы:**
- Один «лишний» байт потребляется до throw (read-then-check) — для DoS несущественно.
- Блочный путь переопределён отдельно, чтобы `Files.copy` не падал в медленный single-byte `read()`.
- Байт `0x00` возвращается как int `0` → `b >= 0` корректно считает его (бинарные файлы).

## Блок 10. Байтовая семантика `read()` vs `read(byte[])`

| Метод | Возврат | Что значит |
|---|---|---|
| `read(): Int` | -1..255 | -1 = EOF, 0..255 = байт unsigned (0 = реальный байт `0x00`) |
| `read(byte[], off, len): Int` | -1..len | -1 = EOF, 1..len = реальный объём, 0 = `len==0` |

**checkLimit построен под оба:** `>= 0` / `> 0` отделяют EOF и `len==0` от реальных данных; байт `0x00` считается (иначе бинарные аплоады имели бы «укороченный» лимит).

**Потолок типа:** `maxBytes` и счётчик — **Long** (~8 EiB), не Integer. `Integer.MAX_VALUE` ≈ 2 GiB мелькает в размере одного чанка и в `ByteArray`, но **не** ограничивает общий размер — копирование стримится через фикс-буфер, накопление идёт в Long. Реальный потолок — диск + Ktor-limits + config + FS.

## Блок 11. Конфигурация — вынос параметров

**Что сделано:** `allowedExtensions` и `maxBytes` — параметры конструктора `FileReceiver` с дефолтами в `companion object`. Класс **уже** config-agnostic — источник значений решает wiring.

**Куда выносить:** `application.conf` (HOCON, уже есть, использует `${?ENV}`-паттерн) + env-override. Значения варьируются по окружениям / меняются бизнес-решением → конфиг, не код.

**Упаковка:** `application.conf` уезжает **внутрь fat-jar** после сборки — не редактируется руками. Это фича: JAR = шаблон с defaults + `${?ENV}`-хуки, реальные значения поставляет окружение при запуске (env / `-D` / `-Dconfig.file`). Один артефакт гоняется через dev/staging/prod без пересборки = 12-factor.

## Блок 12. Gate и tooling

**Реальный gate:** `./gradlew compileKotlin` → `BUILD SUCCESSFUL`. **Не** AFT-диагностики — у bundled Kotlin LSP версия 2.1.0 vs проект 2.3.21 → ложные «incompatible metadata version» ошибки (зафиксировано в `AGENTS.md`).

**Где живут правила:** `./AGENTS.md` (проект, tooling + conventions) и `~/.config/opencode/AGENTS.md` (глобально, стиль). Проектный перекрывает глобальный. После правки — рестарт opencode.

**Проверки Фазы 5:** `grep FileResult` → 0 ссылок; `println` в рефакторенных модулях → 0.

## Архитектурный итог (финальная схема слоёв)

```
FileRouter (транспорт)
  ├─ транспортные предусловия: нет файла → 400 (до домена)
  ├─ .use{} для InputStream (владелец потока)
  └─ маппинг ReceptionResult → HTTP (200/400/422, частичный успех)
        │ ReceptionResult
        ▼
FileReceiver (домен)
  ├─ бизнес-правила: расширение, лимит размера, sanitize-имя
  └─ перевод StorageOutcome → ReceptionResult
        │ StorageOutcome
        ▼
FileStorage (инфра)
  ├─ байты на диск → StorageOutcome
  ├─ узкий IOException catch, CancellationException-проброс
  └─ logger
```

Три типа, три owner'а, три семантики. `NotProvided` ушёл в транспорт. Винегрет закрыт. Слои не смешиваются.

## Принципы для рефлексии (выжимка)

1. **Пустой catch = удалить.** Catch оправдан действием внутри.
2. **`CancellationException` — святое.** Не глотать, не логировать, узкий catch исключает ошибку.
3. **Чистый код ≠ прод-код.** Прод закрывает невидимые риски данных.
4. **Кто открыл, тот закрыл** (RAII, `.use{}`).
5. **Один тип — один слой-создатель, одна семантика.** Смешение слоёв в типе = запах.
6. **Транспортные предусловия — на транспортном слое.** Домен не знает «отсутствие».
7. **Attacker-controlled вход → санитизация.** Имя файла из HTTP = враждебное.
8. **Доверяй реальным байтам, не заголовкам.** DoS-гард считает поток.
9. **Конфиг для того, что варьируется/меняется без релиза.** Технические константы — в код.
10. **Gate — реальная сборка, а не LSP-диагностики** (особенно при metadata-мismatch).
